Interactive AI Study Guides

AI authors a study guide from an audiobook chapter as a list of typed, schedulable cards plus non-schedulable visuals. The reader self-rates each card, and FSRS schedules future quizzes. Cards and visuals are delegated types so we get one uniform query surface and strong typing per kind.

End-to-end flow

1. Chapter

Transcript ready, user opens the chapter.

2. Generate

Job calls the AI with the registry of supported kinds.

3. Validate & store

Each item is saved as a typed row; invalid items drop.

4. Render

Partials key on kind_type; Turbo streams updates.

5. Review

Self-rate → FSRS schedules the next due.

Why delegated types here

Uniform surface

Audiobook::Chapter::Card.all answers "what does this user have?" and "what's due today?" without caring about kind. FSRS state and ordering live on the parent.

Typed specifics

Each kind owns its columns — MultipleChoice#correct_index, Cloze#answers, Ordering#items. No jsonb-with-schema. AR validations do the schema work.

Data model

Two delegated-type hierarchies. Card carries FSRS state; Visual doesn't. Both are ordered inside a guide via a polymorphic study_guide_items join.

Generation pipeline

What the job does, step by step
  1. Build a prompt from the chapter text plus the registry — derived from Audiobook::Chapter::Card.delegated_type_types and the parallel set for Visual.
  2. Call the AI; receive a flat items array, each entry tagged with type (card or visual) and kind.
  3. In a transaction: create the StudyGuide, create each concrete kind row, create the parent Card/Visual pointing at it, append a StudyGuideItem with position.
  4. Items failing AR validation drop. If the guide ends up empty, the generation is marked failed.
  5. Turbo-broadcast completion to the chapter page.
The AI's contract (response shape) 
{
  "items": [
    { "type": "visual", "kind": "diagram",
      "attributes": { "nodes": [...], "edges": [...] } },

    { "type": "card", "kind": "multiple_choice",
      "concept_title": "FSRS state",
      "source_excerpt": "…verbatim passage…",
      "attributes": {
        "question": "Which property…",
        "options": ["Difficulty", "Stability", "Lapses", "Scheduled days"],
        "correct_index": 1,
        "rationale": "Stability defines the forgetting curve."
      }
    }
  ]
}
Why fresh-start on regeneration

Reconciling new cards against old ones via a stable concept_key would preserve schedules across regenerations, but it adds a merge step the AI has to participate in. Fresh-start is simpler: old cards keep their FSRS schedule and continue to surface in the global inbox; they just don't appear in the current guide view. The price is some accumulated redundancy if you regenerate often; we can address that later with an archive action.
